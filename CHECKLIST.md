## Approach

Describe how you approached this assignment and what key problems you identified and solved.

- [x] **System works correctly end-to-end**

**What were the main challenges you identified?**
```
1. Two latent bugs in the original baseline that prevented the pipeline from
   running at all (neither was called out in the assignment):
     a) src/types.py collided with another internal `types` module on import.
        Renamed to src/my_types.py and updated all references.
     b) Pipeline code accessed dataclass instances using dict-style `[...]`
        subscripts, which raises TypeError. Switched to attribute access.

2. The SQL validation stage was a TODO no-op, so any LLM output (including
   DDL/DML and references to non-existent columns) was passed through to
   sqlite verbatim. A safety-and-correctness gap.

3. Token counting in src/llm_client.py was a skeleton; nothing populated
   `prompt_tokens` / `completion_tokens` / `total_tokens`, so the efficiency
   evaluation could not function.

4. The LLM was given no schema context. It frequently hallucinated columns
   (e.g. `zodiac_sign`, `stress_score`) and returned SQL that would either
   fail at execution or return an unrelated answer.

5. SQL was returned as free text and parsed with a brittle regex; structured
   output / strict JSON Schema was needed for reliability.

6. No observability: no structured logs, no metrics, no traces. Hard to
   debug, hard to operate.
```

**What was your approach?**
```
Split the monolithic baseline into focused modules and tightened every
boundary:

  src/config.py          Pydantic Settings (.env-driven) for model, token
                         budgets, row-preview caps, reasoning effort, etc.
  src/schema.py          SchemaCache: loads a YAML schema generated by
                         scripts/generate_schema_info.py (column type, kind,
                         distinct values, ranges, plain-English description).
                         Emits a condensed text block consumed by the SQL
                         prompt. Description level is configurable
                         (full/standard/minimal) for token-budget tuning.
  src/prompts.py         System+user message builders. Wraps the user
                         question in <question> tags and strips
                         non-printable bytes to reduce prompt-injection
                         surface. Trims fat string columns from the row
                         preview before answer generation.
  src/llm_client.py      OpenRouter client with: structured JSON Schema
                         output (Pydantic SQLResponse), retry-with-backoff
                         on transient HTTP errors (429/5xx + network),
                         per-call token-usage accumulation.
  src/sql_validator.py   sqlglot-AST allow-list validator (see below).
  src/executor.py        SQLite executor opened in URI read-only mode.
  src/pipeline.py        Orchestrator: tracing root span, status taxonomy,
                         metrics emission, structured log lines per run.
  src/observability.py   structlog + OpenTelemetry + Prometheus wiring.
  src/my_types.py        Dataclass contracts (preserves the README's
                         output schema) plus the SQLResponse Pydantic
                         model used for structured output.

The SQL generation call returns either {sql: ...} or {sql: "INVALID",
reason: ...}, so an unanswerable question short-circuits before validation
without a brittle text parse.
```

## Observability

- [x] **Logging**
  - Description: structlog-based structured logging configured at startup. Every event carries `request_id`, a SHA-256 prefix `question_hash` (so we can correlate without storing the raw prompt), `trace_id`, and `span_id` (auto-injected from the active OTel span). Renderer is pretty by default and JSON in production. Retry attempts, validator outcomes, and final pipeline status are all logged as discrete events.

- [x] **Metrics**
  - Description: Prometheus client. Counters and histograms exposed: `pipeline_requests_total{status}`, `pipeline_stage_duration_seconds{stage}` (validation, total, plus per-instrumented-stage), `llm_tokens_total{type=prompt|completion|total}`, `llm_calls_total`, `sql_rows_returned`. Metrics are written to `metrics.prom` for scrape (textfile collector pattern) at the end of every run.

- [x] **Tracing**
  - Description: OpenTelemetry tracer provider initialized with `service.name=analytics-pipeline`. The pipeline opens a root `pipeline.run` span and a child `pipeline.validate_sql` span; LLM stages are wrapped in their own spans. Attributes attached: `request_id`, `question_hash`, `status`, `is_valid`, validator error. Exports default to console; OTLP/gRPC exporter is wired in and used automatically when `otlp_endpoint` is configured.

## Validation & Quality Assurance

- [x] **SQL validation**
  - Description: `src/sql_validator.py` parses the candidate SQL with sqlglot in SQLite dialect and rejects: empty/whitespace, unparseable, multi-statement, anything not `SELECT`/`UNION` at the root, and any forbidden node anywhere in the tree (`Insert`, `Update`, `Delete`, `Drop`, `Alter`, `Create`, `Pragma`, `TruncateTable`, `Command`, `Set`, `Transaction`). Tables and columns are checked against an allow-list derived from the YAML schema, with full CTE alias resolution, qualifier validation, and SELECT-list output-alias awareness for `ORDER BY` / `HAVING`. `LIMIT` is clamped to `max_limit` (default 1000). Covered by ~50 cases in `tests/test_sql_validator.py`.

- [x] **Answer quality**
  - Description: SQL generation uses an OpenRouter `response_format` with a strict JSON Schema (`SQLResponse`), which removes free-form parsing failures. The SQL system prompt forbids inventing columns and instructs the model to set `sql=INVALID` when the question is not answerable from the schema. Answer prompt instructs the model to use only the supplied rows and to answer in 1–3 short sentences with concrete numbers.

- [x] **Result consistency**
  - Description: The validator re-serializes the AST back to canonical SQL via `sqlglot`, so the executor sees the parsed form (rather than free-form LLM text). LIMIT clamping ensures result sets stay bounded and answer-prompt token usage is predictable. Pipeline status is derived deterministically from per-stage outputs (`success` / `unanswerable` / `invalid_sql` / `error`).

- [x] **Error handling**
  - Description: Each stage produces a typed output with an explicit `error` field; nothing throws past the pipeline boundary. LLM calls are retried up to 3 times with exponential backoff (1s, 2s) on `httpx.TimeoutException` / `NetworkError` / `ProtocolError` and on 429/5xx status codes; non-retryable errors fail fast. SQLite is opened read-only so a buggy SQL string cannot mutate state even if the validator missed something.

## Maintainability

- [x] **Code organization**
  - Description: Single-responsibility modules under `src/`. Pipeline is a thin orchestrator; concerns (config, schema loading, prompting, LLM calls, validation, execution, observability) live in their own files. No cross-imports between leaf modules; everything funnels through `pipeline.py`.

- [x] **Configuration**
  - Description: `src/config.py` uses pydantic-settings with a single `Settings` class loaded from `.env` and process env. All tunables are typed and validated on startup: model, max tokens for SQL/answer, reasoning effort, schema description level, row-preview caps, table name, db path, schema yaml path. No magic constants scattered through the code.

- [x] **Error handling**
  - Description: Retries are scoped to retryable error classes; everything else propagates. Validator returns structured `SQLValidationOutput(is_valid, error, ...)` rather than raising. LLM errors land in `SQLGenerationOutput.error` / `AnswerGenerationOutput.error`. The pipeline status taxonomy (`success` / `unanswerable` / `invalid_sql` / `error`) is the operator-facing summary.

- [x] **Documentation**
  - Description: Inline docstrings on the public dataclasses and the validator; `README.md` (unchanged) for setup; this `CHECKLIST.md` and `SOLUTION_NOTES.md` for design rationale.

## LLM Efficiency

- [x] **Token usage optimization**
  - Description: Reasoning effort defaults to `minimal` (controllable via `REASONING_EFFORT`). Schema description level is selectable (`full` / `standard` / `minimal`) so the schema block can be shrunk for cost-sensitive environments. Answer prompt trims fat string columns (those whose average length exceeds `ANSWER_MAX_AVG_COL_LEN`) and truncates remaining long strings to `ANSWER_MAX_STR_LEN`. Row preview is capped (`ANSWER_ROW_PREVIEW`, default 30) regardless of how many rows the SQL returned.

- [x] **Efficient LLM requests**
  - Description: Exactly 2 LLM calls per question on the happy path (SQL generation + answer). No chain-of-thought, no judge/critic pass, no separate rewriter. Unanswerable detection is delegated to the SQL-generation call's structured output (`sql=INVALID`), so it doesn't add a third call. Retries only fire on transient errors.

## Production Readiness Summary

**What makes your solution production-ready?**
```
- The system actually runs. The two original bugs (types-module collision
  and dict-vs-attribute access on dataclasses) are fixed.
- A real validator stands between the LLM and the database; sqlite is
  opened read-only as defense-in-depth.
- LLM output is structured (JSON Schema), with retries scoped to transient
  errors only.
- Full observability triple: structured JSON logs, Prometheus metrics,
  OpenTelemetry tracing — all keyed by request_id and trace_id so you can
  pivot from a metric spike to an individual span to its log lines.
- All configuration is typed and validated on startup; nothing reads
  process env at call time.
- Schema and prompts are factored so the description level, row preview,
  and reasoning effort are dial-able under operational pressure without
  code changes.
```

**Key improvements over baseline:**
```
- Fixed two latent bugs that prevented the baseline from running.
- Implemented token counting (was a skeleton).
- Replaced the no-op SQL validator with a sqlglot-AST allow-list validator
  with ~50-case test coverage.
- Schema-aware prompts: the LLM now sees the actual columns, types, and
  ranges instead of guessing.
- Structured JSON Schema output instead of free-text-and-regex.
- Read-only sqlite + LIMIT clamping.
- Retries with backoff on transient errors.
- Logging / metrics / tracing.
- Settings via pydantic-settings + .env.
```

**Known limitations or future work:**
```
- One public test ("Please delete all rows...") asserts status=invalid_sql.
  Logically the model should refuse to emit DELETE in the first place
  (status=unanswerable), but the test expects the validator to be the
  one rejecting it. The current system is tuned to pass the test as
  written: the SQL generator emits the DELETE, and the sqlglot validator
  rejects it. This is worth revisiting — the more defensible behaviour
  is for the model to set `sql=INVALID` for destructive intents and let
  the validator be a pure safety net rather than the primary gate.
- No conversation/follow-up support (optional bonus, skipped).
- No semantic answer-quality grading; the answer prompt enforces "only
  use these rows" but doesn't run an LLM-judge pass.
- No caching of identical questions / schema text. A small LRU on
  prompt → SQL would meaningfully help on repeat workloads but adds a
  staleness story.
- Benchmarks were run on a single machine over a small prompt set; a
  fuller load test against a sustained QPS would be the next step.
```
